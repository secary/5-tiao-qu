# 5-tiao-qu

一个面向桌游玩家的规则查询 App。MVP 阶段先以网页端为主，通过 LLM + RAG 对扫描后的桌游规则书进行问答查询，并尽量让每个回答都能追溯到具体规则来源。

## 产品定位

桌游规则通常页数多、术语密集、例外情况多，玩家在实际游玩时很难快速翻到准确答案。本项目希望做成一个「桌游规则裁判助手」：

- 用户可以上传规则书 PDF、扫描件或拍照图片。
- 系统自动识别规则文本，并建立可检索的规则知识库。
- 用户用自然语言提问，系统基于规则书内容回答。
- 每个回答都返回出处，例如页码、章节、原文片段或页面截图。
- 当规则书中没有明确答案时，系统应提示不确定，而不是编造规则。

## MVP 目标

第一版重点验证「上传规则书 -> 构建索引 -> 查询规则 -> 展示出处」这条主链路。

### 必做功能

1. 规则书导入
   - 支持上传单本桌游规则书。
   - 优先支持 PDF；后续再补充多张图片上传。
   - 对扫描版规则书执行 OCR。
   - 保留页码、章节标题、段落文本和原始页面引用。

2. 规则文本处理
   - 按页码、标题、段落对文本进行切分。
   - 为每个文本块生成 embedding。
   - 将文本块、页码、章节、来源文件、向量索引存储起来。
   - 避免切分过碎导致上下文丢失，也避免切分过大影响检索精度。

3. 规则问答
   - 用户输入自然语言问题。
   - 系统从向量库中检索相关规则片段。
   - LLM 只基于检索结果回答。
   - 回答中必须包含引用来源。
   - 检索结果不足时，返回「规则书中未找到明确说明」。

4. 网页端界面
   - 左侧展示已导入的桌游规则书。
   - 中间提供聊天式问答区域。
   - 右侧展示引用的规则原文、页码和页面预览。
   - 支持连续追问，让用户围绕同一局面继续追问。

5. 答案可信度
   - 区分明确规则、相关规则推断和未找到答案。
   - 显示命中的规则片段。
   - 尽量避免无出处回答。

### 暂不做功能

- 用户账号体系。
- 桌游社区和公开规则库。
- 多人协作房间。
- 复杂权限管理。
- 移动端 App。
- 自动购买或下载官方规则书。

## 核心用户流程

1. 用户进入网页端。
2. 上传一份桌游规则书。
3. 系统解析 PDF 或图片，执行 OCR。
4. 系统展示导入进度：上传、识别、切分、索引完成。
5. 用户输入问题，例如「攻击时可以穿过队友吗？」。
6. 系统检索相关规则片段。
7. LLM 生成简洁回答。
8. 页面展示答案、引用页码、原文片段和可回看的规则页。
9. 用户继续追问，例如「如果角色有飞行能力呢？」。

### 后端

- FastAPI 或 Node.js
- 文件上传与任务状态管理
- OCR 调度
- 文本切分
- embedding 生成
- RAG 检索
- LLM 回答生成

### 存储

- PostgreSQL：存储规则书、页面、文本块、问答记录等结构化数据。
- pgvector / Qdrant / Chroma：存储和检索 embedding。
- 本地文件系统或对象存储：保存原始 PDF、页面图片和 OCR 结果。

### OCR

MVP 可以先选择一种方案快速跑通：

- 本地 OCR：适合成本可控和隐私要求较高的场景。
- 云 OCR：适合早期快速验证识别质量。

OCR 输出建议至少包含：

- `page_number`
- `text`
- `blocks`
- `bounding_box`
- `source_file`

## RAG 查询流程

1. 用户问题进入后端。
2. 对问题生成 embedding。
3. 从向量库检索最相关的规则片段。
4. 按页码、章节和相似度重排结果。
5. 将命中的规则片段和用户问题一起传给 LLM。
6. LLM 生成答案，并标注引用来源。
7. 后端返回：
   - 简洁答案
   - 引用规则片段
   - 页码
   - 章节标题
   - 置信度或回答类型

## 回答格式建议

```text
结论：可以 / 不可以 / 规则书没有明确说明。

依据：
- 第 7 页「移动与阻挡」：……
- 第 12 页「特殊移动」：……

说明：
如果问题涉及扩展规则、FAQ 或勘误，需要优先检查对应资料。
```

## 规则优先级设计

后续导入扩展规则、FAQ 和勘误时，需要为不同来源设置优先级：

1. 官方勘误 / FAQ
2. 扩展规则
3. 基础规则
4. 玩家自定义备注

当多个来源冲突时，系统应明确提示冲突，并展示冲突来源。

## 第一阶段里程碑

1. 初始化网页端项目。
2. 完成单文件上传。
3. 完成 PDF 文本提取或 OCR。
4. 完成规则文本切分。
5. 接入 embedding 和向量库。
6. 完成基础 RAG 问答接口。
7. 完成网页端查询界面。
8. 展示答案引用来源。
9. 使用 1-2 本规则书进行端到端验证。
