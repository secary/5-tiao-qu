# AGENTS.md

本文件给进入这个仓库的各类智能体阅读。请先理解产品目标，再决定代码、文档或架构改动。

## 项目概览

`5-tiao-qu` 是一个面向桌游玩家的规则查询 App。MVP 阶段优先做网页端，通过 LLM + RAG 对上传或扫描后的桌游规则书进行问答查询。

核心价值不是“让模型自由回答规则问题”，而是：

- 基于用户上传的规则书回答。
- 给出可验证的页码、章节、原文片段或页面截图。
- 找不到明确依据时如实说明不确定。
- 帮玩家在实际游玩中快速获得可信规则裁定。

请阅读 `README.md` 获取当前产品设计和 MVP 范围。

## 当前阶段

项目处于早期设计和 MVP 验证阶段。优先级应服务于这条主链路：

1. 上传规则书。
2. 解析 PDF 或图片。
3. OCR / 文本提取。
4. 按页码、章节和段落切分。
5. 生成 embedding 并建立向量索引。
6. 用户自然语言提问。
7. 检索相关规则片段。
8. LLM 基于检索结果回答。
9. 展示答案出处。

除非用户明确要求，不要优先做账号、社区、公开规则库、移动端 App 或复杂权限系统。

## 协作原则

- 默认使用中文书写用户可见文档和界面文案。
- 改动前先查看现有文件和 README，不要凭空引入大型架构。
- 保持 MVP 思路，优先跑通端到端流程。
- 代码和文档都应服务于“规则查询可信、可追溯、可验证”。
- 不要删除或覆盖用户已有改动。
- 如果需要新增框架、数据库、OCR 服务或 LLM 服务，先说明理由和取舍。
- 对扫描规则书、OCR、向量库、LLM 调用等能力，优先设计可替换接口，避免把业务逻辑绑死在单一供应商上。

## RAG 质量要求

桌游规则查询对准确性要求高。实现 RAG 相关功能时，请遵守以下要求：

- 回答必须基于检索到的规则片段。
- 回答必须带来源引用，至少包含页码和片段文本。
- 检索不到足够依据时，应返回“规则书中未找到明确说明”。
- 不要用常识、经验或模型猜测补全规则。
- 保留原始页码、章节标题、来源文件和文本块 ID。
- 支持后续显示原文和页面截图。
- 对基础规则、扩展规则、FAQ、勘误等来源预留优先级字段。
- 当多个来源冲突时，应展示冲突来源，而不是静默合并。

建议将回答类型显式区分为：

- `direct_rule`：规则书中有直接依据。
- `related_inference`：只能从相关规则推断。
- `not_found`：没有找到明确规则。
- `conflict`：不同来源存在冲突。

## 推荐数据对象

早期实现可以围绕这些概念建模：

- `BoardGame`：桌游条目。
- `Rulebook`：规则书文件和版本。
- `RulePage`：页码、页面图片、OCR 状态。
- `RuleChunk`：切分后的规则片段、页码、章节、embedding 引用。
- `Question`：用户提问。
- `Answer`：模型回答、回答类型、置信度。
- `Citation`：答案引用的规则片段。

字段设计时请优先保留追溯信息，例如：

- `source_file`
- `page_number`
- `chapter_title`
- `chunk_text`
- `chunk_index`
- `bounding_box`
- `source_type`
- `source_priority`

## 前端体验方向

MVP 网页端建议采用三栏或近似三栏的信息结构：

- 左侧：规则书 / 桌游列表。
- 中间：聊天式规则查询。
- 右侧：引用来源、原文片段和页面预览。

交互重点：

- 上传时展示导入进度。
- 回答旁边展示引用来源。
- 用户可以点击引用回看原文。
- 连续追问时应保留当前规则书和最近上下文。
- 不要把首页做成营销落地页，优先展示可用的查询体验。

## 后端服务边界

建议将后端拆成清晰的领域模块：

- 文件上传与存储。
- PDF 解析和图片渲染。
- OCR 调度。
- 文本清洗与切分。
- embedding 生成。
- 向量检索。
- RAG prompt 构建。
- LLM 回答生成。
- 引用来源组装。

如果还没有明确技术栈，可以优先考虑：

- 前端：Next.js 或 React。
- 后端：FastAPI 或 Node.js。
- 结构化数据：PostgreSQL。
- 向量检索：pgvector、Qdrant 或 Chroma。
- 文件存储：本地文件系统，后续再替换为对象存储。

## Prompt 与回答约束

构建问答 prompt 时，应明确要求模型：

- 只使用提供的规则片段。
- 不知道就说不知道。
- 用用户提问的语言回答。
- 先给结论，再给依据。
- 每条依据都绑定引用 ID。
- 不输出没有引用支撑的规则裁定。

推荐回答结构：

```text
结论：可以 / 不可以 / 规则书没有明确说明。

依据：
- 第 X 页「章节名」：相关规则片段。

说明：
必要时解释适用条件、例外情况或冲突来源。
```

## 测试与验收

实现功能时，优先验证以下路径：

- 可以上传一份规则书。
- 可以完成解析、OCR 或文本提取。
- 可以生成规则片段和索引。
- 用户问题能检索到相关片段。
- 回答包含可点击或可展示的引用来源。
- 当问题没有依据时不会编造答案。

建议准备 1-2 本规则书作为端到端测试样例，并覆盖：

- 直接命中规则的问题。
- 需要跨页组合的问题。
- 规则书没有明确答案的问题。
- 术语中英文混合的问题。

## 不要做的事

- 不要让 LLM 无检索结果直接回答规则裁定。
- 不要丢失页码、来源文件或原文片段。
- 不要在 MVP 阶段引入过重的社交、账号或运营功能。
- 不要把 OCR、embedding、向量库、LLM 供应商写死到业务层。
- 不要把用户上传的规则书默认公开或共享。
